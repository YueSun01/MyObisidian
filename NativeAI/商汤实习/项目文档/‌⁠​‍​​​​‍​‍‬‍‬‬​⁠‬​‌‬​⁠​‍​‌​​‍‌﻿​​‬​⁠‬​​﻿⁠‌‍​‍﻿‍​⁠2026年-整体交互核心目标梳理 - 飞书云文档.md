# 三、核心-北极星指标

**北极星指标****（响应latency、有效交互时长、能力外延的边界、交互质量** **@孙文秀** **）**

1. **多模态一致性** —— 语音、动作、视觉（含视线）全模态在同一任务链路下保持交互一致对齐语义（对应：统一入口 + 主 Agent 含VLM）

3. **可执行度与工具链及其他****智能体****的协作** —— 任务链路能真实闭环：讲解、动作执行、路径、检索等能被执行且执行正确，同时合理的监控执行状态 配合交互（对应：多 Agent Toolchain）

5. **主动性** —— 系统基于视觉 / 状态判断自动调节节奏、主动发起引导，而不是被动问答（对应：主动交互策略 + 状态感知）
    
6. **长时稳定性** —— 30–60 分钟场景中无漂移、无崩溃、无推理阻

### 交互VLM
    

整体VLM是交互Agent的核心，负责除其他复杂任务执行的日常语音交互反馈；核心只负责交互，基于输入后的解析信息，通过语音提供像人一样的AI自然交互（部分情况会涉及出发图片理解能力，反向索取帧信息，进行识别后回复），但同时交互Agent也需要配合监控各个agent的执行情况，获取相关执行状态，和用户实现交互

交互 VLM 是全系统的“人机交互大脑”，

负责多模态理解、自然语言交互、视觉解释与场景反馈， 并通过 Monitor 协调各任务 Agent 的执行状态，为用户提供流畅、人性化、可信赖的 AI 对话体验。

- 性能与鲁棒性（Performance）

# 四、技术目标

- 场景驱动 → 产品能力 → 技术能力 → 业务指标
    
- 讲解稳定性 → 流式架构稳定 → 低延迟 + 打断恢复 → 提升满意度
    
- 互动深度 → 多模型协同决策 → 意图识别 + VLM融合 → 提升差异化

## **技术目标：围绕“交互入口 × 主 Agent × 多 Agent 工具链 × VLM × Monitor”形成统一的大脑**

技术目标必须从当前的系统架构自然生长出来，而不是独立模块化定义。我们以“用户 → 交互入口 → 主 Agent → 多 Agent 工具链执行 → VLM 状态反馈 → Monitor → 用户”的 **端到端统一大脑** 为核心，将技术目标重构为五个方向：

## **O1：统一的流式交互入口（作为系统大脑的“感知与调度入口”）**

交互入口不再只是“语音入口”，而是整个系统的**统一感知 + 状态聚合 + 调度入口**。它承载了端到端链路的第一跳，决定系统的节奏、稳定性与自然度。

结合当前技术设计，它必须完成：

- **多模态汇聚**：语音、视觉、动作、空间状态统一进入同一 Session
    
- **一体化状态管理**：统一对“当前讲解进度 / 用户状态 / 场景状态”建模
    
- **任务调度****网关**：把用户侧的意图实时路由给主 Agent
    
- **可打断、可恢复、****低延迟****稳定运行**
    
- **支持和 Monitor 的实时互通（心跳、错误****回调****、状态回收）**
    

## **O2：“主 Agent” （意图理解+交互主Agent）的统一上下文理解与任务规划能力**

主 Agent 是整个系统的大脑，负责：

- 意图理解
    
- 状态融合
    
- 规划任务链
    
- 调用多 Agent 工具链
    
- 汇总执行结果
    
- 驱动下一轮交互
    

结合你前面的设计，主 Agent 技术目标包括：

- **统一上下文模型**：用户状态、场景状态、讲解节点、动作进度
    
- **统一任务图（Task** **Graph****）生成**：不是模块堆叠，而是一次任务的整体规划
    
- **与监控系统（Monitor）联动**：主 Agent 必须能处理异常、恢复链路、监控各个任务情况，并集合执行状态 驱动交互
    
- **流式推理（Incremental Reasoning）**：能边讲边看边调整
    

## **O3：多 Agent 工具链协同执行（把系统从语言助手提升为可互动交互的，完善****人机交互****的执行的****智能体****）**

- **动作 / 讲解 / 检索 / 控制工具链的协同执行**（不是单点调用）
    
- **同一任务图下的串并行调度**
    
- **执行结果的结构化返回，供主 Agent 决策下一步**
    
- **动作 – 文本/语音 – 视觉反馈三者一致**
    
- **可持续扩展更多工具，不改架构**
    

## **O4：VLM 驱动的可解释状态反馈（系统外显能力的关键差异点）**

主交互Agent“VLM 负责把机器内部任务执行 → 自然地反馈给用户，是系统差异化优势。”

对应的技术目标应为：

- **VLM 负责生成可解释反馈**：任务执行 → 场景感知 → 用户可理解的自然表达
    
- **把内部执行图变成自然语言 + 动作的外显形式**
    
- **让系统的动作、讲解、路径规划都具备视觉一致性**
    
- **让用户“看到系统如何决策、为什么这样做”**
    
- **支持复杂场景：讲解指向、****兴趣点****捕捉、视线跟随**
    

系统里区分友商的最关键能力之一,作为独立技术目标。

## **O5：Monitor × 回归 × 系统稳定性的端到端能力（系统进入规模化的前提）**

**整个系统具备健康检查、****回调****、状态管理、恢复机制。**

技术目标要明确：

- **所有 Agent 的健康状态都可监控（心跳、状态漂移）**
    
- **交互链路可恢复、可重试**
    
- **可回放、可回归、多模态一致性评测**
    
- **场景稳定性****压测****（≥30分钟连续讲解）**
    
- **稳定性结果反哺到技术指标体系**
    
- **对业务线输出“可评测、可复用、可交付”的整体系统能力**

**用户输入 → 交互入口(O1) → 主 Agent 决策(O2) → 多 Agent 工具链执行(O3) → VLM 可解释+交互反馈(O4) → Monitor 稳定性闭环(O5) → 用户**

# 五、业务目标

## **O1：在“讲解 + 交互”场景实现体验级差领先，取得行业市占率**

讲解是 2026 年交互方向的主战场，需要以真实业务场景的体验差异确立领先地位，而不是靠 Demo 或参数对比。 目标是在**博物馆 / 展厅 / 政企展馆**等重点场景形成：

- 主动交互显著优于同类机器人与大模型解决方案
    
- 动作 / 语音 / 视觉+机器人视线 三模态表现自然一致
    
- 节奏、内容结构、反馈机制达到“可长期稳定服务”的产品标准
    
- 以真实落地的场景和案例推动市场心智建立（而非纯技术亮点）
    

核心验证方式：

- 拿到 **市占率 / 落地数量 / 场景覆盖** 的可量化指标
    
- 获得行业用户和 B 端客户的可感知体验反馈
    
- 场景中长时稳定运行（无明显漂移和行为突变）
    

业务定义的成功：**在目标重点场景内建立稳定、可复用、可推广的体验领先心智，形成公司级标杆案例。（实现市占率TOP-3）**

## **O2：让交互系统成为内部产品线的统一底座能力（公司级平台化）**

交互能力不仅是单一业务线的能力，而是公司多条业务线（数字人、教育、硬件载体、终端设备）的共用基础。

业务目标要求：

- 交互系统需以平台化形式提供能力输出
    
- 通过统一方便的接入方式，让多个产线“拉通接入”
    
- 无需重新搭建交互链路（ASR / TTS / VLM / Agent / Monitor 直接复用）
    
- 成为公司内部使用交互能力的首选，而不是其中一个选项
    
- 交互能力的“跨业务一致性”需要可量化可交付（稳定性、速度、控制力）
    

平台化衡量标准：

- **≥2 条内部产品线**基于交互底座落地真实场景（**覆盖全数字人实时交互场景**）
    
- 交互系统成为**默认集成方案**（无需二次搭建）
    
- 新场景适配成本明显下降（业务自行拼装，而不是底层重构）
    

业务定义的成功：**交互系统从“一个业务模块”升级为“基础设施级平台”，成为内部多业务增长的核心底座。**

## **O3：通过智能硬件的规模覆盖验证平台能力，而不是依赖单点营收**

交互系统需要绑定智能硬件落地（机器人 / 终端设备 / 展厅载体等），以规模化的方式验证平台能力的“通用性 + 稳定性 + 可扩展性”。

业务目标要求：

- 覆盖不同硬件形态（机器人、智慧屏、数字人终端、轻型可移动设备）
    
- 在多类硬件上跑同一交互底座（少量适配 / 保持一致体验）
    
- 以“覆盖数量和兼容度”验证真正的平台能力
    
- 不以营收为第一目标，而是以“场景规模扩展”作为验证重点
    
- 为后续公司级 toB / toG / toC 智能硬件布局提前打底
    

验证标准：

- 交互系统可在**≥3 类硬件形态**稳定运行
    
- 不同设备之间的适配成本可控（以配置为主，不以重开发为主）
    
- 形成“硬件 + 交互”的完整产品体验链路
    

业务定义的成功：**交互系统真正成为“硬件行业的默认选择”，通过规模覆盖验证平台的可复制性与可扩展性。**
